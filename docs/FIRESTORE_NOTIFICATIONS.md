# Firestore Notifications Database Structure

## Collection Schema

### Collection: `notifications`

#### Document Structure

```typescript
{
  id: string;                    // Auto-generated by Firestore
  userId?: string;               // Optional - specific user ID for personal notifications
  role: string;                  // admin | collector | user
  title: string;                 // Notification title
  message: string;               // Notification body/content
  type: string;                  // info | warning | alert | AI-suggestion
  status: string;                // unread | read
  createdAt: Timestamp;          // Server timestamp when created
}
```

## Notification Types

- **info**: General informational messages
- **warning**: Important warnings requiring attention
- **alert**: Critical alerts requiring immediate action
- **AI-suggestion**: AI-generated recommendations or insights

## Security Rules Explanation

### Rules Location
Place the rules in `firestore.rules` file and deploy via Firebase Console or CLI:
```bash
firebase deploy --only firestore:rules
```

### Security Model

1. **Authentication Required**
   - All read/write operations require authentication
   - Users must be logged in via Firebase Auth

2. **Role-Based Access Control (RBAC)**
   - Users can only read notifications where:
     - `role` field matches their authenticated role
     - OR `userId` field matches their UID (personal notifications)
   - Prevents users from reading other roles' notifications

3. **Read Permissions**
   - ✅ Users can read their role's broadcast notifications
   - ✅ Users can read personal notifications sent to them
   - ❌ Users cannot read other roles' notifications

4. **Create Permissions**
   - ✅ Authenticated users can create notifications (typically admin only)
   - Enforces data validation:
     - All required fields must be present
     - Type must be one of: info, warning, alert, AI-suggestion
     - Status must be: unread or read
     - userId is optional (null for broadcasts)

5. **Update Permissions**
   - ✅ Users can only update the `status` field (marking as read/unread)
   - ✅ Only applies to notifications they can read
   - ❌ Cannot modify title, message, type, or other fields

6. **Delete Permissions**
   - ✅ Users can delete only their personal notifications (userId matches)
   - ❌ Cannot delete broadcast or other users' notifications

## Implementation Notes

### Custom Claims for Role-Based Security
Ensure your Firebase Auth custom claims include user role:
```typescript
// When creating/updating user, set custom claim:
await admin.auth().setCustomUserClaims(uid, { role: 'admin' });
```

The security rules reference `request.auth.token.role`, so this must be set.

### Indexing
No composite indexes required for basic queries. Single-field queries are automatically indexed.

For advanced queries (e.g., filtering by role AND status), consider:
```typescript
// Add composite index for queries like:
query(notificationsRef, where('role', '==', 'admin'), where('status', '==', 'unread'))
```

### Data Growth Considerations
- Use Firestore's TTL (Time-to-Live) policy to auto-delete old notifications
- Consider archiving read notifications after 30 days
- Implement pagination for large notification lists

### Querying Examples
```typescript
// Get unread notifications for a user's role
const q = query(
  collection(db, 'notifications'),
  where('role', '==', userRole),
  where('status', '==', 'unread')
);

// Client-side filtering for personal notifications
const personalNotifications = notifications.filter(
  n => !n.userId || n.userId === currentUserId
);
```
